PROTOCOL — answering "how do cards A and B interact?" (follow ALL steps). STEP 0 — CACHE CHECK: look for ix-<card-a>-x-<card-b> (slugs alphabetical; ls notes/ | grep '^ix-'). Found AND CR build matches third_party/fab-rules/VERSIONS.txt → answer FROM IT, but a cached answer is NEVER final on its own — CACHED-ANSWER FLOW (mandatory): (a) answer immediately FROM the note, LABELING it clearly as a previously-adjudicated cached ruling (state its adjudication date + CR build — the user must know this is a prior answer, not fresh work); (b) in the same message tell the user you will now double-check the interaction just in case (fast answer first, confirmation afterwards); (c) re-run verification (steps 1b, 5, 6 minimum: live true text, CR section cites, rulings_errata) and report whether the cached ruling still holds or what changed; (d) proceed to STEP 9 human validation. CR bumped → re-verify (steps 2-6) and RE-MINT with the new stamp. Then: (1) CARD TEXT: card-vault brain first, fallback corpus/live per ROLE.md; note SETS frontmatter + keywords. (1b) TRUE TEXT (MANDATORY for precision-sensitive rulings): `fab-cli fabtcg card "<name>"` — Card Vault true text (authority per CR 2.0.2) + printed diff + errata; true text WINS over corpus/vault notes; update the card note on divergence. (2) KEYWORDS: recall [[kw-*]] for every keyword via [[keywords-index]]. (3) RELEASE NOTES: recall release-notes-<set> for EACH set either card belongs to + [[release-notes-index]]; their frontmatter lists cards:/keywords: — `grep -l "<card-slug>" notes/release-notes-*.md`. (4) COMBAT/TIMING notes when the chain is involved. (5) VERIFY against en-fab-cr.txt with section cites; refresh if >24h. (6) OFFICIAL RULINGS: `fab-cli fabtcg card "<name>" --json` → rulings_errata. (7) HONESTY: cited vs derived separated; CR-silent → UNSETTLED → #ask-a-judge. STEP 8 — STORE: mint/re-mint the ix- note (CR stamp in source, UNSETTLED inline, links to [[kw-*]] + release-notes notes) ending with a CONFIDENCE line: "CONFIDENCE: cr-verified" (default) | "human-confirmed <date>" | "unsure-pending-confirmation" | "disputed — do not answer from this note". STEP 9 — HUMAN VALIDATION LOOP (orchestrator, after relaying a ruling to the human — ALWAYS, including cached answers, and only AFTER the double-check of the cached-answer flow has run): use AskUserQuestion WITH a preview of the ruling's core line so the human can annotate/add notes, options: Correct / Incorrect / Unsure. "Correct" → re-mint with CONFIDENCE: human-confirmed <date>, and on a cached note APPEND a reaffirmation timestamp (keep a running "reaffirmed: <ISO datetimes>" trail in the note — confidence grows with each confirmation; repeated confirmations → graduation candidate; fold any user notes from the answer into the note). "Incorrect" → set CONFIDENCE: disputed, then RE-RUN the full protocol from scratch; the human's objection is treated per the knowledge-flow rule like a player suggestion — investigate against the documents; if after re-research you cannot see where the ruling went wrong, ASK the user why it is wrong before re-minting; escalate to #ask-a-judge if the documents don't settle it, and only re-mint a corrected ruling once confirmed; never leave a disputed note answerable. "Unsure" → CONFIDENCE: unsure-pending-confirmation; research deeper AND/OR ask the user which part they didn't understand or doubt (their annotation notes often say); seek external confirmation (Card Vault rulings_errata, #ask-a-judge) and upgrade/downgrade on the outcome; do not graduate unconfirmed notes. INVALIDATION: on new CR build or set ingest, grep ix-* for changed keywords and re-verify. Examples: [[ix-fleeing-starbreeze-x-gone-in-a-flash]], [[ix-command-and-conquer-x-uzuri-switchblade]].
