PROTOCOL — answering "how do cards A and B interact?" (follow ALL steps). STEP 0 — CACHE CHECK: look for an existing ix-<card-a>-x-<card-b> note (slugs in alphabetical order; ls notes/ | grep '^ix-'). If found AND its recorded CR build matches third_party/fab-rules/VERSIONS.txt, answer FROM IT (it's a verified ruling); if the CR has bumped since, re-verify its claims (steps 2-6) and RE-MINT it with the new CR build stamp (strength bumps, created preserved). Then: (1) CARD TEXT: card-vault brain first (card-<slug>-<color>.md), fallback corpus/live per ROLE.md; note each card's SETS frontmatter and keywords. (2) KEYWORDS: recall [[kw-*]] for EVERY keyword on both cards via [[keywords-index]]. (3) RELEASE NOTES (the step that gets skipped): recall release-notes-<set> for EACH set either card belongs to PLUS [[release-notes-index]] (supersession + cross-set principles). (4) COMBAT/TIMING: [[combat-chain-steps]], [[combat-step-rulings]], trigger/priority/LKI notes when the chain is involved. (5) VERIFY precision-sensitive claims against third_party/fab-rules/en-fab-cr.txt with section cites; refresh docs if >24h. (6) OFFICIAL RULINGS: cardvault.fabtcg.com for both cards. (7) HONESTY: separate cited from grammar-derived conclusions; CR-silent points marked UNSETTLED → #ask-a-judge ([[escalation-ask-a-judge]]) — never guessed. STEP 8 — STORE THE RESULT: mint/re-mint the ix- note with the full verified ruling: CR build stamp in the source field, every UNSETTLED point marked inline, links to the involved [[kw-*]] and [[release-notes-<set>]] notes (so recall reaches it from keyword OR set angles). INVALIDATION: whenever a new CR build or set release-notes are ingested, grep ix-* notes for the changed keywords/sections and re-verify each hit (same as the release-notes supersession pass).
